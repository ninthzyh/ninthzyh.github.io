nginx

nginx

1.简介

http、反向代理、静态页面的服务器；

支持动态语言（不支持Java）；

高负载支持高达5w并发；

2.正向代理

客户端直接无法访问网址(www.google.com)，就先在浏览器中配置代理服务器，用代理服务器去访问这个网址。

3.反向代理

目标服务器看作不同后端的地址

客户端不会直接访问目标服务器（8080、8081、8082），客户端（不需要在浏览器中配置代理服务器）先访问反向代理服务器（9001），反向代理服务器再去访问目标服务器获取数据，然后再把数据返回给客户端。

对外暴露反向代理服务器的地址，隐藏目标服务器的地址，对用户来说访问的是（反向代理+目标服务器）一台服务器。

4.负载均衡

反向代理的目标服务器有多个，分散一台服务器的请求、相应的压力。

单一服务器：

客户端请求服务端数，服务端操作数据库，获取到了数据相应客户端

多个服务器：上图反向代理

5.动静分离

反向代理服务器把请求动态资源和静态资源分在两台服务器上

6.nginx命令

查看开放的端口号：
firewall-cmd --list-all

linux防火墙开放端口号：
sudo firewall-cmd --add-port=8001/tcp --permanent

重启防火墙：
firewall-cmd --reload

/usr/local/nginx/sbin

./nginx -v

查看nginx状态：
ps -ef | grep nginx

启动：
./nginx

关闭：

7. 反向代理

1）实例1

客户端网址输入www.123.com:8080 打开tomcat欢迎页

1.启动tomcat（bin/startup.bat）地址：localhost:8080
2.windows/System32/drivers/etc/hosts修改映射，将本机ip【目标服务器ip】映射到www.123.com名称上【如www.wisdom-eye.com】
192.168.1.49 www.123.com 访问www.123.com就相当于访问本机
【目标服务器ip www.wisdom-eye.com访问www.wisdom-eye.com相当于访问目标服务器】
3.启动nginx进入nginx欢迎页localhost:80，客户端访问反向代理服务器，代理服务器将请求转发到目标服务器tomcat的欢迎页【代理服务器将请求转发到目标服务器的ip上】
4.修改nginx配置（config/nginx.conf）
监听81端口，这是反向代理暴露出来的端口，server块中proxy_pass配转发的目标服务器地址localhost:8080【项目目标地址】
--------------------------------------------------------------------------
server{
		listen		 81;
		server_name  localhost;
		location / {
			root html;
			proxy_pass  http://127.0.0.1:8080;
			index  index.html  index.htm;
		}
}
5.测试，访问www.123.com:8080跳转到tomcat的欢迎页
2）实例2

客户端http://localhost:8080访问反向代理服务器9009端口，反向代理服务器分别请求

http://36.137.0.237:81/images/

http://36.134.212.144:82/live/

http://36.134.212.143:83/audio/

然后把返回数据给客户端

server {
        listen       9009;
        server_name  localhost;		

        location /qnxm/ {
            proxy_pass    http://36.137.0.237:81/qnxm/;
        }
        # 匹配路径是/image/，访问http://36.137.0.237:81/images/
		location /images/ {
			proxy_pass    http://36.137.0.237:81/images/;
		}
		# 匹配到路径中有live--->访问http://36.134.212.144:82/live/
		location ~ /live/{
			proxy_pass    http://36.134.212.144:82/live/;
		}
		# 匹配到路径中有audio--->访问http://36.134.212.143:83/audio
		location ~ /audio/{
			proxy_pass    http://36.134.212.143:83;
		}
        location / {
            proxy_pass    http://localhost:8080/;
        }
        error_page   500 502 503 504  /50x.html;
        location = /50x.html {
            root   html;
        }
        
    }

8.location配置

location匹配

优先级从高到低

location = /a{} = /a 精准匹配a，完全相同
location ^~ /a{} ^~ /a 以a开头
location ~ /a{} ~ 区分大小写，正则匹配，路径中有a
location ~* /a{} ~* 不区分大小写，正则匹配，路径中有a
location !~ /a{} !~ 区分大小写，正则匹配，路径中没有a
location !~* /a{} !~* 不区分大小写，正则匹配，路径中没有a
location /a{} / 通用匹配，任何请求都会匹配到

匹配

1）优先级 从高到低= ^~ ~ /

2）优先级相同，匹配度高的先匹配

3）匹配度也相同，按先后顺序

/a /a/b /a/d /d
location = /a{} 1 - - -
location ^~ /a{} - 2匹配优先级^高于~ 3 -
location ^~ /a/b{} - 1匹配度高于 ^/a - -
location ~ ^/[a-z]{} - 3优先级相同按先后顺序 1 1优先级相同按先后顺序
location ~ ^/\w{} - 4 2 2

反向代理

1）proxy_pass的ip后有/，访问地址ip+请求路径（不加loaction后面的匹配字符）

2）proxy_pass的ip后没有/，访问地址ip+loaction后面的匹配字符+请求路径

location /a{
	proxy_pass		http://ip;
}
location /b/{
	proxy_pass		http://ip/;
}
--------------------------------------------------------------------------
等价于
/a/x	--->	通过/a访问http://ip/a/x
/b/x 	--->	通过/b/x/访问http://ip/x
alias别名

alias后的路径不加location的匹配，alias后路径以 / 结尾；

root后的路径不管location匹配的是什么 都要拼在路径后；

# 如果访问 www.domain.com/i/test.png 最后返回的结果为 /data/w3/i/test.png ,root 配置最后的/要不要都行
location /i/ {
    root /data/w3/;
}

# 如果访问 www.domain.com/i/test.png 最后返回的结果为 /data/w3/test.png ,root 配置最后的/一定是要的，否则返回结果会变成  /data/w3test.png  而返回的404
location /i/ {
    alias /data/w3/;
}
server root || location root

有location / {}优先指向里面的root，如果没有location \ {}，就会执行server root

# location /{}有root，优先指向D:/environment，如果没有则指向C:/environment

server {
        listen       80;
        server_name  localhost;
        root  C:/environment;
		autoindex on;
		location / {
            root   D:/environment;
            index  index.html index.htm;
        }
    }

9.负载均衡

浏览器访问http://localhost/a/，平均请求到36.137.0.237的81和82端口

upstream group 配置两个目标服务器，客户端访问http://localhost/a/时，因为访问权重都是1，所以跳转的几率相同。权重越高，分配客户端越多 

worker_processes  1;
events {
    worker_connections  1024;
}
http {
    include       mime.types;
    default_type  application/octet-stream;
    upstream group{
    	server http://36.137.0.237:81 weight=1;
    	server http://36.137.0.237:82 weight=1;
    }
server {
        listen       9009;
        server_name  localhost;	
        
		location /a/ {
			proxy_pass    http://group/;
		}
		
        error_page   500 502 503 504  /50x.html;
        location = /50x.html {
            root   html;
        }
        
    }
}

负载均衡策略

1）轮询：随机分配到目标服务器

2）权重：根据权重大小分配到目标服务器

3）ip_hash：upstream中加ip_hash，同一ip第一次分配到了哪个服务器，后面访问都会到这个服务器上

4）fair：根据响应时间分配服务器，时间短的优先分配

10.动静分离

动态请求：数据库的操作。

静态请求：请求资源文件。

1）动态请求和静态请求分别放在两个服务器。

2）动态请求和静态请求混合在一个服务器上，用nginx分开。

location指定不同后缀实现不同的转发，expires设置过期时间。如3d，三天后发送请求对比服务器文件最后更新时间，时间没变化返回304，不会从服务器抓取数据；如果有变化返回200，重新从服务器下载数据进行缓存。

静态文件请求，不管location后的匹配和root最后有没有/，都把浏览器的访问路径端口号后和location root后的路径拼在一起

静态文件请求：root

动态文件请求：proxy_pass

# http://localhost:81/Desktop/1.jpg  C:/Users/pc/Desktop/1.jpg
server{
		listen		 81;
		server_name  localhost;
		location /Desktop {
			root C:/Users/pc;
			index  index.html  index.htm;
		}
	}

11.高可用

浏览器发送请求到一台nginx上，一台nginx可能宕机。

两台nginx服务器做主、备服务器，当主服务器nginx1宕机，就用备份服务器nginx2去请求目标服务器。对外浏览器请求是一个虚拟ip，虚拟ip默认绑定在主服务器上，主服务器宕机服务器上的keepalived会将虚拟ip绑定到备份服务器上，就能继续请求目标服务器。

1）主备服务器上需要安装keepalived软件

2）高可用配置在etc/keepalived.conf，配置主备服务器

3）/usr/local/src添加检测服务器是否宕机的脚本

4）启动两台服务器和上面的keepalived（systemctl start keepalived.service）

5）浏览器中访问虚拟ip会把请求发送到主服务器nginx1，主服务器宕机就会把请求发送到备份服务器nginx2

12.原理

查看nginx进程

ps -ef | grep nginx

nginx有1个master（管理员）多个worker（工人），master接到任务发给worker，worker争抢任务。worker的数量由服务器cpu数决定，保证每个worker都有一个cpu可用，在nginx.conf的全局块中设置worker_processes n;

1）发送请求占用worker几个连接数：请求静态资源2个（client<—>worke），动态资源4个（client<—>worker<—>tomcat1）。

2）最大并发数：worker_processes*worker_connection（最大连接数1024）/一个请求的连接数（静态请求2/动态请求4）

3）1个mastern个worker的好处：

  • ​ 每个worker是独立的，一个worker出现问题，其他worker继续工作不会造成服务中断；
  • 热部署后老的worker在请求处理完后被kill，新的请求会被没有任务的worker争抢。
